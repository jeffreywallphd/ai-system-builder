# Secret Master-Key Re-encryption Operations

This runbook documents Story 8.3.6 (Feature 8 / Epic 8.3): controlled re-encryption of stored active secret versions after a KEK/master-key change.

## Purpose

- Re-wrap existing active secret material under the currently active master key without forcing manual secret re-entry.
- Provide restartable, progress-tracked, auditable operations with explicit administrative control.

- `src/application/security/use-cases/ReEncryptSecretsUseCase.ts`

## Operational prerequisites

- Existing and new key material must both be resolvable during migration:
  - `AI_LOOM_SECRET_MASTER_KEY_ID` must reference the new active key.
  - Old key ids/values must remain available through `AI_LOOM_SECRET_MASTER_KEYS_BY_ID` until the operation completes.
- Secret service encryption must be configured and healthy (`/api/v1/security/secrets/health`).
- Caller must use an authenticated trusted session.

## API workflow

Start or resume operation:

- `POST /api/v1/security/secrets/maintenance/re-encryption`
- Body:
  - `operationKey` (optional; autogenerated when omitted)
  - `operationId` (optional; provide to resume a previously failed/running operation)
  - `maxTargetsPerInvocation` (optional; default `50`, max `500`)
  - `occurredAt` (optional ISO-8601 timestamp)

Read operation status:

- `GET /api/v1/security/secrets/maintenance/re-encryption/:operationId`

## Progress and state model

Each operation persists:

- immutable target list of active secret versions at operation start,
- status: `running`, `succeeded`, or `failed`,
- counters: total/processed/succeeded/failed/remaining targets,
- failure entries with `secretId`, `versionId`, and reason/message,
- timestamps and optimistic `revision` for safe concurrent updates.

Operations are resumable by calling start endpoint again with the same `operationId` and `operationKey`.

## Failure and recovery posture

- Per-target metadata drift (missing secret/version or concurrent version change) is recorded as target failure and processing continues.
- Decrypt/encrypt/persistence exceptions move the operation to `failed` with persisted partial progress and safe fixed last-error metadata (raw exception text is not persisted).
- Recovery path:
  1. Correct key/configuration or persistence issue.
  2. Resume operation with the same `operationId` + `operationKey`.
  3. Verify terminal `succeeded` state via status endpoint.

## Audit and safety notes

- Secret audit events record both `secret.access-decision` and `secret.operation` for `re-encrypt`.
- Observability event `secret.re-encrypt` records operation outcome and progress details.
- Plaintext and encrypted payload internals are not exposed through API responses.
